# P9.2 — Offline-first PWA (service worker, directory cache, app-shell)

**Status:** 📝 Spec stub (planned). Part of milestone `v1.3.0-engagement-loop`.
**Branch (planned):** `feat/offline-pwa`
**Sibling phases:** P9.1 (hymne, ships first), P9.3 (WhatsApp Phase A, ships last)

## A. Origin

The audience profile makes offline-first a material UX win, not a vanity feature. Members are mobile-first on low-end Android with patchy data; reopening the directory shouldn't cost them a network round-trip every time. Today the site is a vanilla server-rendered Django app with no service worker, so every directory visit costs ~50 KB + image fetches even when the data hasn't changed.

A small but well-scoped PWA layer transforms three frequent flows from "wait + spend data" to "instant + free":

- Reopening the annuaire on the bus.
- Looking up a colleague's WhatsApp number outside coverage.
- Returning to a memory page when bandwidth craters.

The infrastructure cost is small: the manifest already exists (added in the favicon phase), no new app needed, and the directory dataset is tiny (~50 KB JSON for 200 members).

## B. Goals

1. Members who have visited at least once can browse the **annuaire** offline (search, filter, view cards) with no network.
2. Static assets (CSS, JS, fonts, app-shell HTML) are served from cache after first visit, eliminating render-blocking round-trips.
3. The PWA installs to home screen on Android (manifest already supports this, but `display` upgrades from `minimal-ui` to `standalone`).
4. The existing `data_saver` notification preference is respected: image prefetch never runs when set.
5. A clear **offline indicator** in the UI when navigating without connectivity, plus a clean fallback page for routes not in cache.

## C. Non-goals (explicit)

- **No offline writes / sync queue.** Member edits, magic-link reissues, cooptation actions all require connectivity. Offline = read-only.
- **No background sync API.** Browser support is uneven; not worth the complexity at our scale.
- **No push notifications.** WhatsApp is the notification channel; web push adds a permission prompt for zero benefit.
- **No offline support for memoires/memoriam image content.** Those are Cloudinary-served images; rely on browser HTTP cache + lazy load. Trying to prefetch all photos would defeat the data-saver goal.
- **No new app or model.** Pure additive frontend + one read-only JSON endpoint.
- **No PWA-specific UX divergence on desktop.** Desktop browsers get the same service worker but no install prompt.

## D. Audience

Same as platform. Particular concentration on the data-saver-aware subset (members with `NotificationPreference.data_saver=True`). The PWA layer is **transparent** for users who don't install — service worker still caches, just no home-screen icon.

## E. Architecture

### E.1 Manifest upgrade

`static/manifest.webmanifest` already exists (lines from current state):

```json
{
  "name": "Les Retrouvailles",
  "short_name": "Retrouvailles",
  "display": "minimal-ui",
  "start_url": "/",
  "lang": "fr",
  "theme_color": "#7a3b2e",
  "background_color": "#fdf6e3"
}
```

Changes:
- `display`: `minimal-ui` → **`standalone`** (full app feel; URL bar hidden when installed).
- Add `scope`: `"/"` (PWA covers entire site).
- Add `categories`: `["social", "lifestyle"]` (helps Android Play Store-style listings).
- Add maskable icon variants if not already present (check `static/img/`; add a 512×512 maskable variant if missing — generated via the existing `scripts/generate_favicons.py`).

### E.2 Service worker

New file `static/js/sw.js`. Registered from `templates/base.html` via a small inline script:

```js
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => navigator.serviceWorker.register('/sw.js'));
}
```

Service worker served from `/sw.js` (root scope) by a tiny Django view in `core/views.py` that returns the static file with the right MIME and cache headers (`Service-Worker-Allowed: /`). Can't serve from `/static/js/sw.js` because of the scope rule (SW scope ≤ its serving path).

**Cache strategies:**

| Resource type | Pattern | Strategy | TTL |
|---|---|---|---|
| App shell HTML (`/`, `/annuaire/`, `/aide/`, `/guide/`, `/hymne/`, `/profile/<slug>/`, key static views) | `network-first` falling back to cache | network-first | session |
| Tailwind output.css, vendored HTMX, app-shell JS | `cache-first` | cache-first | until version bump |
| `/api/annuaire.json` directory data | `stale-while-revalidate` | SWR | revalidate on every visit |
| Cloudinary images | Pass-through (browser HTTP cache handles it) | network-only | — |
| All other URLs | network-only (no caching) | network-only | — |

**Versioning:** SW cache name includes a build version constant (e.g., `retrouvailles-v1`). Bumping the version in `sw.js` invalidates all caches on next load. Doc this in `guide_admin.md` so operators know to bump on release.

### E.3 Directory data endpoint

New endpoint `GET /api/annuaire.json` in `members/views.py` (or `members/api.py` if cleaner separation wanted). Returns the same dataset the existing `directory_view` reads, serialized to JSON:

```json
{
  "version": "2026-05-08T17:30:00Z",
  "members": [
    {
      "slug": "abc-123-...",
      "first_name": "Aïssa",
      "last_name": "S.",
      "nickname": "",
      "city": "Niamey",
      "country": "Niger",
      "profession": "Professeure retraitée",
      "years_attended": [1981, 1982, 1983, 1984],
      "classes": ["6e", "5eA", "4eA", "3eA"],
      "photo_url": "https://res.cloudinary.com/.../w_120,h_120,c_fill/...",
      "show_email": true,
      "show_whatsapp": true,
      "show_city": true
    }
  ]
}
```

- Member-only (default LOGIN_REQUIRED middleware).
- Respects the same `show_*` flags as the HTML directory.
- `version` is a timestamp so the SW cache can detect "newer than what I have" without diffing rows.
- Capped at 500 rows hard limit (200 today, headroom for future).
- Cache-Control: `private, max-age=60` so even without SW, a multi-tab session reuses the response.

### E.4 Frontend offline UX

- **Offline indicator banner.** When `navigator.onLine === false`, show a slim warm-tinted banner at the top: "Mode hors-ligne — l'annuaire reste accessible." Auto-dismiss when back online.
- **Fallback page for uncached routes.** If a member navigates to a route the SW hasn't cached (e.g., `/gestion/` while offline), serve a `templates/core/offline.html` page with a clear message and a link to the annuaire.
- **Install prompt.** Capture `beforeinstallprompt` event, surface a "📱 Installer l'app" button in the navbar (or footer) on mobile only, only after the user has visited at least 2 times (localStorage flag). Suppress aggressively — never auto-prompt.

### E.5 Data-saver respect

Existing `NotificationPreference.data_saver` flag is currently un-wired in templates. As part of this phase:

- Surface the flag in a Django context processor so templates can read `request.member.preferences.data_saver`.
- The directory JSON endpoint emits `photo_url: null` for the data-saver case (server-side decision, not client-side).
- Image lazy-loading: `<img loading="lazy">` everywhere already by HTML default; reaffirm in directory cards.
- Service worker explicitly skips Cloudinary fetches when the request originates from a data-saver session (check via custom header set by middleware).

## F. Locked decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| 1 | Service worker location | **Served from `/sw.js`** via a tiny Django view | SW scope rule requires root path. |
| 2 | Cache strategy for directory | **Stale-while-revalidate** | Instant render from cache, freshness on every visit. |
| 3 | Cache strategy for static | **Cache-first with version bump** | Avoid revalidation cost on hot assets. |
| 4 | Manifest `display` mode | **`standalone`** | True app-feel when installed. |
| 5 | Install prompt | **Manual button, gated by 2+ visits** | Avoid auto-prompt fatigue. |
| 6 | Offline writes | **Out of scope** | Sync queues add complexity; WA is the offline-write channel. |
| 7 | Data-saver image strategy | **Server-side `photo_url: null`** | One source of truth; no client trust needed. |
| 8 | Cache versioning | **Manual bump in `sw.js`** | Operator-friendly; no build pipeline change. |
| 9 | Push notifications | **Out of scope** | WhatsApp is the channel. |
| 10 | API endpoint location | **`members/views.py`** initially | Promote to `members/api.py` if more endpoints follow. |

## G. Open questions

1. **Cache invalidation on member edits.** When a member updates their profile, the cached annuaire JSON is stale until SWR catches it on next visit. Acceptable for v1? (Yes — directory isn't real-time critical.)
2. **Profile detail offline?** First version: skip caching `/profile/<slug>/` HTML. Members who clicked into a profile while online get it via app-shell HTML cache anyway. Revisit if usage signals it matters.
3. **Storage budget.** SW cache should soft-cap around ~5 MB per device (sufficient for 200 rows + small static assets). Aggressive eviction policy if exceeded?

## I. Risks

- **iOS Safari quirks.** PWA support on iOS is improving but still flaky (e.g., display modes, install banners). **Mitigation:** explicit iOS Safari manual QA pass; degrade gracefully to "regular website" on iOS.
- **Cache poisoning if `Vary` header wrong.** If we serve different content to data-saver vs non-data-saver users at the same URL, the SW cache could mix them. **Mitigation:** include the data-saver flag in the cache key (separate cache buckets).
- **First-paint regression.** A poorly written SW can delay first paint on first visit. **Mitigation:** SW registers AFTER `load` event; first visit is unaffected.
- **Update lag.** Members may stay on a stale SW version for days. **Mitigation:** `clients.claim()` + skipWaiting on install for fast version takeover; doc the version-bump procedure.
- **Storage quota exhaustion on low-end Android.** Some devices give PWAs only ~10 MB. **Mitigation:** soft-cap cache; oldest-first eviction; logging via `navigator.storage.estimate()`.

## J. Out of scope (deferred)

- Background sync for offline writes.
- Push notifications.
- Periodic background sync for prefetch.
- Workbox or other SW frameworks (vanilla JS is sufficient at our scale).
- Offline support for `gestion/` (admin can be online-required).
- Translation cache for future Hausa locale.
